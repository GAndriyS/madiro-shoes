# Madiro — аналіз вимог

**Застосунок обліку взуття для магазину (Львів)**
Джерело: design handoff з Claude Design (`design_handoff_shoe_inventory`, експорт 16.07.2026) — README, tech stack, три дизайн-файли (`План та архітектура`, `Варіації сканера`, `Дашборд адміна`).
Статус: базовий документ для розробки. Дизайн — high-fidelity: кольори, типографіка, відступи, копірайтинг і стани фінальні.

---

## 1. Загальний опис

**Призначення.** Облік взуття для одного фізичного магазину: поступлення, продаж, списання, повернення, залишки та аналітика (виручка/маржа). Ключова особливість — на коробках рукописні бірки з трьома наліпками **SIZE / COLOR / STYLE** (усі значення числові, напр. `38 / 36 / 7645`); замість штрихкодів використовується **фото бірки + vision-LLM**, який розпізнає рукописні цифри. Людина завжди підтверджує результат розпізнавання.

**Контекст і масштаб.**
- Один магазин, кілька продавців, один адміністратор.
- Точна кількість пар невідома — інтерфейси й API мають витримувати **сотні–тисячі позицій**.
- Багато старого товару **без відомої ціни закупки** — система підтримує пари без вхідної ціни; статистика веде такий товар окремим рядком.
- Валюта — ₴ (грн). Мова — українська, з перемикачем на англійську (лише на екрані логіну).

**Два клієнти на спільному бекенді:**
1. **PWA-сканер** (мобільний, mobile-first) — щоденна робота в магазині: продаж/списання, поступлення-чернетки, повернення, ручний ввід, пошук по складу. Користувачі: продавці та адмін.
2. **Веб-дашборд адміна** (адаптивний: десктоп/планшет/телефон) — огляд і статистика, склад, черга поступлень «Очікують ціни», користувачі. Доступ: лише адмін.

**Порядок реалізації (зафіксований у дизайні):** PWA-сканер → веб-дашборд → (фаза 2) публічна вітрина.

## 2. Ролі та права

Дві ролі: `admin` та `seller`. Спільний акаунт для сканера й дашборда (один логін/пароль); дашборд доступний лише адміну.

### Адміністратор
- Повний доступ: дашборд + сканер.
- Поступлення пар із ціною закупки (або явно «без ціни — старий товар»).
- Підтвердження чернеток поступлення від продавців — з ціною або без.
- Вихід пар (продаж / списання), повернення.
- Керування користувачами, довідником моделей, цінами закупки.
- Може **редагувати й видаляти будь-які записи** — завжди з діалогом підтвердження.
- Бачить усю статистику, включно з маржею.
- **Адміністратор один** і не відображається у списку користувачів; його не можна видалити через UI.

### Продавець
- Сканує пари **на вихід**: продаж (кінцева ціна, готівка/картка) та списання (без ціни, з опційним коментарем).
- Сканує **поступлення як чернетку** — без ціни закупки; пара одразу на складі й доступна для продажу; адмін пізніше підтверджує.
- Може редагувати й видаляти **лише власні чернетки** у статусі «очікує ціни» — з підтвердженням.
- Оформлює повернення від покупця.
- Бачить **лише свої продажі** (за день/місяць, інформаційно, без відсотків і бонусів).
- **Не бачить:** ціни закупки, маржу, загальну аналітику, чужі продажі.

### Матриця доступу (ключові операції)

| Операція | Адмін | Продавець |
|---|---|---|
| Продаж / списання / повернення | ✔ | ✔ |
| Поступлення з ціною закупки | ✔ | — |
| Поступлення-чернетка без ціни | ✔ (може і так) | ✔ |
| Підтвердження чернеток («Ціна» / «Без ціни») | ✔ | — |
| Редагування/видалення будь-яких записів | ✔ | лише власні чернетки «очікує ціни» |
| Ціни закупки, маржа, загальна аналітика | ✔ | ✖ (приховано і в UI, і в API) |
| Керування користувачами | ✔ | ✖ |
| Веб-дашборд | ✔ | ✖ |

> **Життєвий цикл продавця:** продавця не деактивують, а **видаляють** (з підтвердженням). Його продажі, чернетки та поступлення лишаються у звітах — видаляється лише обліковий запис.

## 3. Доменна модель

### 3.1. Сутності

**Style** — код моделі з бірки; **текстове поле** (напр. `7882`). Продавці знають шифр моделей.

**Варіант** = `Style` + `Color` (числовий код з бірки, напр. `36`) + опційно `Material` (шкіра / замша, вручну) + опційно `Season` (без утеплення / байка / овчина, вручну).
- `Season` — внутрішня назва поля в БД; **у всіх інтерфейсах лейбл «Утеплення»**.
- **Ціна закупки зберігається на варіанті** (одна на всі розміри), може бути відсутня.

**Пара** = Варіант + `Size` — **одиниця обліку** (не «кількість на SKU»); кожна пара має власну історію.
- Розміри — **лише цілі числа** (половинних не існує; клавіатура ручного вводу без десяткової крапки).
- Атрибути: дата поступлення, статус, (при виході) кінцева ціна продажу, спосіб оплати (готівка/картка), хто продав, дата виходу. Роздрібна ціна «до знижки» **не зберігається**.
- Статуси: `на складі` (включно з «очікує ціни») → `продана` / `списана`; повернення повертає пару на склад.

**Операції (історія руху)** — продаж / повернення / списання / поступлення, з датою, автором, сумами. У записі поступлення **ціна закупки фіксується на момент поступлення** — пізніша зміна ціни варіанта не переписує історію.

**Користувачі** — продавці (ім'я, логін, пароль, роль завжди `seller`) + один адмін поза списком.

### 3.2. Ідентичність пари та пошук

Унікальність/пошук пари визначають **5 полів**: `Size · Color · Style` (обов'язкові, з бірки) + `Season · Material` (опційні, вручну). Це покриває вимогу «та сама модель може бути на шкірі або замші, утеплена або ні».

### 3.3. Ключові доменні правила (зафіксовані рішення)

1. **Ціна закупки — одна на варіант**, не на пару. Зміна застосовується до всіх пар варіанта; в історії руху лишається історична ціна на момент поступлення.
2. **Чернетка продається одразу:** пара зі статусом «очікує ціни» — на складі й доступна для продажу. Якщо її продано до вказання ціни — продаж реєструється, маржа не рахується, поки нема вхідної ціни (у черзі така картка підсвічується окремо).
3. **Товар без ціни («старий товар»):** адмін може лишити пару без вхідної ціни назавжди — маржа по ній не рахується, статистика веде її окремим рядком (сума і кількість проданого без вхідної ціни).
4. **FIFO:** якщо скан/пошук знаходить кілька ідентичних пар — береться найстаріша.
5. **Неоднозначність варіанта при продажу:** скан дає 3 поля з 5; якщо на складі кілька варіантів (різні утеплення/матеріал) — показувати на вибір **лише реально наявні** комбінації. Окремого екрана немає — це рівень логіки.
6. **Повернення:** ≈14 днів — орієнтир, **без блокування** пізніших повернень. На екрані показується, скільки днів минуло з продажу. Кейс кількох ідентичних проданих пар обробляється найпростіше — береться останній продаж. Продаж скасовується у статистиці (мінусує виручку й маржу).
7. **Повернення чернетки-продажу:** якщо повертається пара, продана у статусі «очікує ціни», — статус відновлюється до «очікує ціни».
8. **Списання — без ціни;** замість поля ціни — опційний текстовий коментар (причина: брак / зразок / втрата). Помилкове списання виправляє **лише адмін** через редагування записів (свідоме рішення).
9. **Підказка ціни продажу:** підставляється остання ціна продажу цього варіанта (збіг усіх 5 полів); поле завжди редаговане.
10. **Підказка ціни закупки:** при поступленні відомого варіанта підставляється ціна минулого поступлення.
11. **Бейдж черги «Очікують ціни» = кількість варіантів** (дія одна на варіант); текстом дублюється «N пар · M варіантів».

## 4. Функціональні вимоги

Ідентифікатори: `FR-S-*` — PWA-сканер, `FR-D-*` — дашборд, `FR-B-*` — бекенд/наскрізні.

### 4.1. PWA-сканер (mobile-first; дизайн: «Варіації сканера», тури 1–6)

**FR-S-01. Вхід.** Логін/пароль; перемикач мови УКР/EN (тільки тут). Підказка: «Облікові записи створює адміністратор. Забули пароль — зверніться до нього» (self-service відновлення пароля немає). Ролі визначають доступ.

**FR-S-02. Головна-хаб.** Привітання з ім'ям і датою; дві великі дії: «Продаж / вихід» і «Поступлення»; кнопка «Повернення від покупця»; картки «Ввести вручну» та «Пошук по складу»; блок «Мої продажі сьогодні» (N пар, сума, перехід до списку). Іконка профілю.

**FR-S-03. Профіль.** Ім'я, роль, логін; посилання «Мої продажі», «Мої чернетки поступлень» (з лічильником); «Вийти з акаунта» (наступний продавець входить зі своїм логіном). Для адміна — перехід до дашборда.

**FR-S-04. Камера.** Видошукач з рамкою під бірку, ліхтарик; фолбек `<input capture>`. Доступ з флоу продажу і поступлення; з камери можна перейти до ручного вводу.

**FR-S-05. Розпізнавання бірки.** Фото → бекенд-пайплайн (стиснення sharp до ~1024px → vision-LLM зі structured output → Zod-валідація) → `{size, color, style}` + впевненість AI. Результат завжди підтверджує людина.

**FR-S-06. Підтвердження скану (вихід).** Фото бірки + редаговані поля SIZE/COLOR/STYLE; опційний вибір «Утеплення · Матеріал» (для звуження до наявних комбінацій); блок «На складі — доступні розміри». Далі — деталі виходу.

**FR-S-07. Деталі виходу — продаж.** Картка знайденої пари (стиль · колір · розмір, матеріал/утеплення, «на складі з …»); тип виходу: Продаж / Списання; ціна продажу (передзаповнена останньою ціною варіанта, редагована); оплата: Готівка / Картка; кнопка «Підтвердити продаж · N ₴». Успіх — повернення на головну з тостом («Продаж зареєстровано, 7645 · 36 · р. 38 — 2 850 ₴ · картка») й оновленим підсумком дня.

**FR-S-08. Деталі виходу — списання.** Без ціни; опційний коментар (напр. «Дефект підошви — повернули на фабрику»); текст «Без ціни — пара списується зі складу»; підтвердження.

**FR-S-09. Помилка «Пару не знайдено».** Якщо збігу на складі немає — явна помилка «Немає пари 9031 · 14 · р. 39. Перевірте поля вище — можливо, AI помилився у цифрі»; блок «Схожі на складі» (близькі збіги з кількістю пар); дії: «Пошук по складу вручну», «Сканувати ще раз».

**FR-S-10. Помилка «Не вдалося прочитати бірку».** Нечітке фото/частково розпізнані цифри: «Піднесіть телефон ближче, увімкніть ліхтарик або введіть значення вручну»; дії: «Спробувати ще раз», «Ввести вручну».

**FR-S-11. Поступлення — адмін.** Камера → підтвердження полів → «Утеплення» і «Матеріал» (опційно, вручну — на бірці їх немає) → «Ціна закупки» (передзаповнена минулим поступленням варіанта) **або** чекбокс «Без ціни — старий товар» → кнопки «На склад і сканувати наступну» (партійний режим для ящика коробок) та «Зберегти й завершити».

**FR-S-12. Поступлення — продавець (чернетка).** Ті самі поля, але без ціни: пояснення «Ціну закупки додасть адміністратор — пара потрапить у чергу "Очікують ціни". Продавати її можна одразу»; кнопки «У чернетки і сканувати наступну» та «Зберегти й завершити».

**FR-S-13. Мої чернетки поступлень.** Список чернеток продавця зі статусами «ОЧІКУЄ ЦІНИ» / «НА СКЛАДІ» (після підтвердження адміном); чернетку «очікує ціни» можна редагувати/видалити — з підтвердженням.

**FR-S-14. Повернення від покупця.** Доступно всім ролям. Скан бірки → система знаходить продаж → картка продажу (пара, ціна, «Продано 12.07 — 2 дні тому · картка · Оля», матеріал/утеплення) → пояснення «Пара повернеться на склад, продаж буде скасовано у статистиці» → «Повернути на склад · −N ₴». Орієнтир 14 днів, без блокування.

**FR-S-15. Ручний ввід пари.** З головної, з камери та з екранів помилок: SIZE/COLOR/STYLE з цифрової клавіатури (без десяткової крапки) + опційно утеплення/матеріал → «Знайти пару» → звичайні деталі виходу.

**FR-S-16. Пошук по складу.** Пошук за Style → список варіантів: колір, матеріал/утеплення, наявні розміри, позначки «N очікує ціни». **Довідковий перегляд** (без переходу в деталі виходу); чернетки враховано як «на складі».

**FR-S-17. Мої продажі.** Лише свої; перемикач «Сьогодні / [поточний місяць]» + вибір іншого місяця з історії через календар; групування по днях; підсумок (пар продано, сума); повернення у списку мінусує суму («−2 700 ₴ · пара знову на складі»). Інформаційно, без бонусів.

**FR-S-18. PWA.** Manifest + service worker (vite-plugin-pwa), installable на домашній екран.

### 4.2. Веб-дашборд адміна (дизайн: «Дашборд адміна», секції 1–4)

Навігація: **Огляд → Склад → Поступлення (бейдж = варіанти в черзі) → Користувачі**.

**FR-D-01. Логін.** Центрована картка, поля ЛОГІН/ПАРОЛЬ, перемикач УКР/EN. Той самий акаунт, що й у сканері; доступ лише для ролі admin.

**FR-D-02. Огляд — KPI.** Перемикач періоду: Сьогодні / Тиждень / Місяць / Період (довільний). Чотири картки: ВИРУЧКА (з порівнянням «+18% до вчора»), ПРОДАНО («8 продажів · 1 повернення · нетто 7 пар»), МАРЖА ДНЯ («45% від виручки»), темна акцентна картка ОЧІКУЮТЬ ЦІНИ («5 пар / від 2 продавців →» — перехід у чергу).

**FR-D-03. Огляд — чарт.** Бар-чарт виручки за 14 днів (сьогодні виділено).

**FR-D-04. Огляд — віджет черги.** Останні чернетки з кнопками «Ціна» та позначкою проданих-без-ціни; оновлюється в реальному часі.

**FR-D-05. Огляд — стрічка останніх операцій.** Час, пара, тип (продаж/повернення/поступлення), продавець, оплата, сума, маржа (повернення — з від'ємною маржею; чернетка — «без ціни»).

**FR-D-06. Склад — таблиця варіантів.** Рядок = варіант. Колонки: STYLE (сортування) / КОЛІР / МАТЕРІАЛ·УТЕПЛЕННЯ / РОЗМІРИ (чипи; бурштиновий чип «2 очік. ціни») / ПАР / ЗАКУПКА (посилання «вказати», якщо нема) / ОСТ. ПРОДАЖ / ДІЇ (редагувати, видалити). Залишок 1 підсвічується. Пошук; чипи-фільтри: Усі / Шкіра / Замша / Овчина / Байка / Без утеплення / «Очікують ціни · N» / Залишок ≤ 2 / Розмір ▾. Пагінація («Показано 6 із 57 · ← 1 2 3 →»). Клік по рядку → картка варіанта.

**FR-D-07. Картка варіанта (шторка праворуч, 520px).** Заголовок «7645 · колір 36»; 3 міні-KPI: ЗАКУПКА / ОСТ. ПРОДАЖ / ПРОДАНО ЗА 30 ДН; список пар поштучно (розмір, «на складі з …», дії редагувати/видалити); історія руху (продаж/повернення/поступлення з історичною ціною закупки в записі).

**FR-D-08. Редагування пари / вказання ціни закупки — одна модалка.** Read-only картка варіанта (поля варіанта не редагуються) + велике поле «ЦІНА ЗАКУПКИ — ОДНА ДЛЯ ВСІХ РОЗМІРІВ ВАРІАНТА» + примітка про застосування до всіх пар («2 пари в черзі — ціна застосується до обох»); з черги кнопка «Зберегти — додати на склад»; наступні поступлення варіанта підставлятимуть цю ціну.

**FR-D-09. Видалення пари.** Модалка підтвердження, «Дію не можна скасувати».

**FR-D-10. Склад — порожній пошук.** «Нічого не знайдено за «7649»» + кнопка «Очистити пошук і фільтри».

**FR-D-11. Поступлення — черга «Очікують ціни».** Згруповано **по варіантах**: картка = варіант, чипи розмірів, кнопки «Вказати ціну» і «Без ціни — старий товар». Особливий кейс: пара продана до вказання ціни — картка з жовтою рамкою, чип «р. 40 · продана», текст «продано за 4 100 ₴ — маржа не рахується, поки нема вхідної ціни». Заголовок: «5 пар очікують ціни · 3 варіанти». **Нових поступлень з дашборда немає** — тільки зі сканера PWA (примітка в заголовку).

**FR-D-12. Поступлення — історія.** Список поступлень з пагінацією («Показано 5 із 34») + підсумок за місяць.

**FR-D-13. Порожня черга.** Зелена галочка, «Нова чернетка зі сканера з'явиться тут автоматично».

**FR-D-14. Підтвердження «Без ціни — старий товар».** Модалка «Лишити без вхідної ціни?» з поясненням наслідків (маржа не рахуватиметься, окремий рядок статистики), кнопка «Так, без ціни».

**FR-D-15. Користувачі.** Список лише продавців: аватар-ініціали, ім'я, логін, бейдж «Продавець», активність («31 продаж за липень · 2 чернетки в черзі», «активна сьогодні 14:32»), дії редагувати/видалити. Адмін — поза списком.

**FR-D-16. Додати користувача.** Ім'я, Логін, Пароль; роль завжди «продавець» (інформаційний блок, вибору ролі немає).

**FR-D-17. Редагувати користувача.** Ім'я, Логін, «НОВИЙ ПАРОЛЬ — залишити без змін» (опційно); внизу посилання «Видалити користувача».

**FR-D-18. Видалення користувача.** Підтвердження: «Її продажі, чернетки та поступлення лишаться у звітах — видаляється лише обліковий запис».

### 4.3. Бекенд / наскрізні

**FR-B-01. Автентифікація.** Passport + JWT; єдиний акаунт для обох клієнтів; авторизація за ролями (guards). Дашборд-ендпоінти — тільки admin.

**FR-B-02. Приховання чутливих даних на рівні API.** Ціни закупки, маржа, аналітика, чужі продажі **не повертаються** продавцю жодним ендпоінтом (не лише ховаються в UI).

**FR-B-03. Vision-пайплайн.** Multipart upload → sharp (~1024px) → vision-LLM (Gemini Flash / GPT-4o-mini / Claude Haiku) зі structured output → Zod-валідація → `{size, color, style}` + впевненість. Інтерфейс `VisionProvider` через NestJS DI — заміна провайдера без зміни коду.

**FR-B-04. Realtime.** Нова чернетка зі сканера автоматично оновлює чергу/бейджі/стрічку на дашборді (Socket.io Gateway; websocket або поллінг — на розсуд реалізації, дизайн вимагає лише «автоматично»).

**FR-B-05. Аналітика.** Виручка й маржа за період; топ моделей і розмірів; залишки по моделях/розмірах (що закуповувати); окремий рядок: сума та кількість проданого без вхідної ціни. Повернення мінусують виручку й маржу періоду.

**FR-B-06. Аудит-цілісність.** Видалення користувача не каскадує на операції; історичні ціни в операціях незмінні; деструктивні дії — тільки з підтвердженням.

## 7. Прогалини та відкриті питання

Те, чого дизайн і план **не покривають** — вирішити до/під час реалізації:

1. **Офлайн-режим PWA.** Service worker заявлений для installability, але поведінка без інтернету (продаж офлайн? черга синхронізації?) не специфікована. *Пропозиція: фаза 1 — тільки graceful-повідомлення «немає з'єднання», без офлайн-черги.*
2. **UI редагування операцій адміном.** Права заявлені («адмін редагує будь-які записи», «помилкове списання виправляє лише адмін через редагування записів»), але екрана редагування операцій (продажу/списання) у дизайні немає — є лише редагування пари/ціни та історія руху в шторці. Потрібно домалювати або зафіксувати мінімальний варіант (напр., скасування операції з історії руху).
3. **Вибір vision-провайдера.** Дизайн допускає Gemini Flash / GPT-4o-mini / Claude Haiku; треба обрати основний (і, бажано, фолбек), оцінити вартість на фото та якість на рукописних цифрах; зберігати чи ні самі фото бірок (retention, місце зберігання) — не визначено.
4. **Websocket vs polling.** Лишено «на розсуд»; стек містить Socket.io — прийняти його як рішення і зафіксувати перелік подій (нова чернетка, підтвердження ціни, продаж).
5. **Відновлення пароля адміна.** Продавцям пароль скидає адмін; для самого адміна механізму немає (тільки доступ до БД/CLI-скрипт?).
6. **Довідник моделей.** У плані згадано «керування довідником моделей» і екран «Довідник моделей» для дашборда, але в фінальному дизайні дашборда секції немає (є лише Склад). Уточнити: чи довідник = похідне від складу (варіанти створюються поступленням), чи потрібен окремий CRUD.
7. **Бекапи та експорт.** Вимог щодо резервного копіювання БД і експорту звітів (CSV/Excel) немає — для реального магазину бекапи критичні.
8. **Ліміти й валідація.** Максимальний розмір фото, діапазон розмірів взуття (напр. 16–50?), максимальна ціна — не специфіковано; додати у Zod-схеми.
9. **Часова зона й «день».** Статистика «за сьогодні» — за місцевим часом магазину (Europe/Kyiv); зафіксувати на бекенді.
10. **Конкурентність.** Два продавці одночасно продають останню ідентичну пару (FIFO) — потрібна транзакційна блокировка на рівні БД.
11. **EN-переклади.** Перемикач EN є, але англійського копірайтингу в дизайні немає — потрібен переклад усіх текстів.
12. **Сесії/термін дії JWT** — не специфіковано (пропозиція: refresh-токени, довга сесія на пристроях магазину).

## 8. Рекомендований порядок реалізації

За планом з дизайну («Починаємо з PWA-сканера»):

1. **Фундамент:** монорепо (Turborepo + pnpm), `shared` (типи + Zod), NestJS-скелет, Prisma-схема, docker-compose з Postgres, auth (JWT, ролі), CI.
2. **PWA-сканер (MVP щоденної роботи):** вхід → головна → ручний ввід + пошук по складу → поступлення (адмін/чернетка) → vision-пайплайн і камера → продаж/списання зі станами помилок → повернення → мої продажі/чернетки.
3. **Дашборд:** логін і каркас навігації (3 брейкпоінти) → Склад (таблиця, шторка, модалки) → Поступлення (черга + realtime) → Огляд (KPI, чарт, стрічка) → Користувачі.
4. **Інженерне доведення:** Playwright e2e критичних флоу (продаж, чернетка→ціна, повернення), SonarCloud gate, деплой (Vercel + Railway/Fly.io).
5. **Фаза 2:** публічна вітрина на Next.js.
