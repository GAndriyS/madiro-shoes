# Madiro — аудит після завершення дашборда (липень 2026)

**Стан на момент аудиту:** змерджені ПР #1–8. Реальні: автентифікація (JWT access+refresh, argon2, ролі), CRUD продавців, PostgreSQL + 2 міграції, CI (lint → format → typecheck → test → build), 38 тестів. На MSW-моках: Огляд, Склад, Поступлення. Наступна фаза — PWA-сканер.

**Методика:** три незалежні перевірки по коду в `main` — безпека бекенду, якість/безпека фронтенду, інфраструктура й масштабування. Знахідки дедупліковано; кожна має файл, серйозність, пояснення і напрям виправлення. Окремо зафіксовано, що **не** є проблемою, щоб не плодити фальшивих тривог.

**Топ-5 пріоритетів:** див. [розділ 6](#6-пріоритезований-бек-лог).

## 1. Ризики безпеки

### 1.1. Високі

**S-H1. Немає rate-limit на логіні + argon2 як підсилювач DoS**
`apps/api/src/main.ts` (жодного throttler-а в проєкті), `apps/api/src/auth/auth.service.ts:32`.
Публічний `POST /api/auth/login` не обмежений ані за IP, ані за акаунтом. Брутфорс нічим не стримується, а кожна спроба з існуючим логіном запускає дорогий `argon2.verify` — неавтентифікований відвідувач може навантажити CPU сервера.
*Виправлення:* `@nestjs/throttler` (напр., 5–10 спроб/хв/IP) + лічильник невдалих спроб на акаунт із тимчасовим блокуванням.

**S-H2. `RolesGuard` «відкритий за замовчуванням» (fails open)**
`apps/api/src/auth/guards/roles.guard.ts:16-18`.
Без метаданих `@Roles(...)` guard повертає `true` — будь-який автентифікований користувач (включно з продавцем) має доступ. Зараз `@Roles('ADMIN')` стоїть лише на `UsersController`. Це міна сповільненої дії: перший же ендпоінт цін/статистики, доданий без явного `@Roles`, **мовчки віддасть продавцю закупівельні ціни й маржу** — пряме порушення ключової вимоги (розділ 2 аналізу вимог).
*Виправлення:* зробити ADMIN безпечним дефолтом для не-`@Public` маршрутів, або додати тест/лінт, що падає на контролерах без явної ролі.

**S-H3. JWT-токени (включно з refresh на 30 днів) у localStorage**
`apps/dashboard/src/stores/auth.ts:13-24` (zustand persist, ключ `madiro.auth`).
Будь-який XSS або скомпрометована залежність зчитує обидва токени; викрадений refresh дає **30 днів** доступу, який нічим не відкликається (див. S-M1). Разова XSS перетворюється на тривале захоплення акаунта адміністратора.
*Виправлення:* refresh — у httpOnly-cookie, access — лише в пам'яті (без persist). Як мінімум — скоротити TTL refresh і додати відкликання.

### 1.2. Середні

**S-M1. Refresh-токени без ротації і без відкликання**
`apps/api/src/auth/auth.service.ts:47-68`, TTL `30d` у `.env.example`.
Stateless JWT без `jti`/серверного реєстру: витік токена неможливо відкликати інакше, ніж зміною `JWT_REFRESH_SECRET` (розлогінює всіх).
*Виправлення:* `tokenVersion` на користувачі або таблиця виданих refresh-ідентифікаторів; ротація при кожному refresh; інвалідація при logout.

**S-M2. Зміна пароля не інвалідовує чинні сесії**
`apps/api/src/users/users.service.ts:44-58` + `jwt.strategy.ts:24-33`.
Валідація перевіряє лише `deletedAt: null`. Після скидання пароля скомпрометованого акаунта старі access-токени живуть до 15 хв, а refresh — усі 30 днів.
*Виправлення:* `passwordChangedAt`/`tokenVersion` у `User`, відхиляти токени, видані раніше.

**S-M3. Слабкий поріг JWT-секретів, placeholder-и проходять валідацію**
`apps/api/src/config/env.validation.ts:7-8` (`min(16)`), `.env.example` (`change-me-access`).
Деплой із секретами-заглушками нічим не блокується.
*Виправлення:* `min(32)`, явна заборона відомих placeholder-ів і збігу access==refresh секретів.

**S-M4. Timing-based перебір логінів**
`apps/api/src/auth/auth.service.ts:32`.
Для неіснуючого логіна відповідь миттєва (без argon2), для існуючого — повільна: за часом відповіді можна дізнатися, які логіни існують.
*Виправлення:* dummy-verify фіксованого хеша на гілці «не знайдено».

**S-M5. CORS відкритий для всіх**
`apps/api/src/main.ts:8` — `enableCors()` з дефолтами (`*`).
Bearer-токени пом'якшують CSRF, але будь-який сайт може викликати API. Стане критичним, якщо колись перейдемо на cookie (див. S-H3).
*Виправлення:* allowlist origin-ів дашборда/сканера через конфіг.

**S-M6. Дані попередньої сесії не чистяться при logout**
`apps/dashboard/src/components/layout/AppNav.tsx:52-55`, `routes/login.tsx`.
`clearSession()` не чистить кеш React Query: наступний користувач за цим же браузером миттєво бачить закешовані адмінські дані (KPI, користувачів) до завершення рефетчів.
*Виправлення:* `queryClient.clear()` при logout і після нового логіну.

**S-M7. Гонка паралельних refresh**
`apps/dashboard/src/lib/api.ts:24-30`.
Кілька одночасних 401 (типова ситуація — сторінка стріляє кількома запитами) запускають кілька паралельних `POST /auth/refresh`. З появою ротації (S-M1) перший успішний інвалідує решту → хибні розлогіни. А якщо refresh поверне некоректне тіло, `authResponseSchema.parse` кидає виняток повз `clearSession()`.
*Виправлення:* single-flight (спільний in-flight проміс refresh) + try/catch навколо parse.

## 2. Ризики масштабування

Вимога: «сотні–тисячі позицій» (аналіз вимог, розділ 1).

**M-1. Патерн мок-`queryStock` не можна портувати в реальний API.**
`apps/dashboard/src/mocks/stock-data.ts` на кожен запит матеріалізує **всі** рядки (`variants.map(toRow)`) двічі: для фільтрації і для summary. Для моку — байдуже; для Postgres — це full scan + агрегація на кожне натискання фільтра.
*Дизайн реального API:* `summary` (пари/моделі/сума закупки) — окремий агрегатний запит (`COUNT/SUM ... GROUP BY`), `items` — окремий пагінований запит по індексах; `sizes`/`awaitingPriceCount` — груповані підзапити по існуючому `@@index([variantId, size, status])`.

**M-2. Унікальність варіанта з NULL — гонка при конкурентному поступленні.**
`schema.prisma` (Variant): `@@unique([style, color, material, season])` + коментар, що NULL-и Postgres вважає різними, і «сервісний шар доперевірить». Сервісна перевірка — це check-then-act: два продавці, які одночасно сканують той самий варіант без матеріалу/утеплення, створять дублікати.
*Виправлення:* Postgres 15+ підтримує `NULLS NOT DISTINCT` на unique-констрейнті — перенести гарантію в БД (Prisma поки що потребує ручного SQL у міграції). Вирішити ДО реалізації поступлень у API.

**M-3. KPI Огляду не повинні сканувати всі операції.**
Контракт `overviewResponseSchema` — один запит на весь екран (це добре для UX), але «виручка/маржа за місяць» наївно рахується сканом таблиці операцій, що зростає щодня.
*Виправлення:* rollup-таблиця денних агрегатів (виручка, маржа, продажі, повернення на день); «сьогодні» — інкрементально; місяць = сума 30 рядків. Закласти в дизайн API до того, як сканер почне генерувати операції.

**M-4. Кешу немає — і поки не треба, але рішення варто зафіксувати.**
Для одного магазину досить rollup-таблиці і правильних індексів; Redis — передчасно. Фіксуємо це тут, щоб не ретрофітити під тиском.

## 3. Надійність фронтенду

Наскрізна тема: **будь-який збій → мовчазний порожній екран**. Всі маршрути рендерять контент за умовою `data &&`, у проєкті нуль `isError`/`ErrorBoundary`/`Suspense`, у `fetch` немає таймаутів, помилки мутацій у модалках (ціна, видалення, «без ціни») ковтаються без повідомлення.

## 5. Що свідомо ОК (перевірено, тривоги не потрібні)

- Видалений продавець втрачає доступ **негайно**: і strategy, і refresh перечитують користувача з БД (`deletedAt: null`), роль читається свіжою на кожен запит.
- Mass-assignment відсутній: поля мапляться явно, роль примусово `SELLER`, адміна через CRUD не створити й не видалити.
- Закупівельні ціни/маржа **поки** ніде не віддаються продавцю — але лише тому, що цінових ендпоінтів ще немає; захист тримається на S-H2.
- Turbo-граф (`typecheck/test/build` → `^build`), tsup dual-format для shared, єдиний lockfile, `allowBuilds`-список, tsconfig-успадкування — коректні.
- CI без секретів; у git немає нічого зайвого (`.env` не трекається).
- Prisma v6 (v7 вийшла) — свідомий вибір; апгрейд окремою задачею.
- Zod v4 у lockstep across shared/api/dashboard — так і тримати.
